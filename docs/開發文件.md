## 1. 專案概述

### 1.1 目標
建立一個 PPT（投影片）線上瀏覽與編輯平台。使用者透過網頁觀看由管理員/使用者建立的投影片簡報，支援角色權限、密碼保護、以及完整的瀏覽追蹤與 IP 記錄。

### 1.2 核心需求摘要
| 編號 | 需求 | 說明 |
|------|------|------|
| R1 | 線上瀏覽投影片 | 使用者可在線上逐頁觀看簡報，支援簡單的上一頁/下一頁切換 |
| R2 | 三種角色 | 管理員（Admin）、使用者（User）、訪客（Guest） |
| R3 | 管理員身分 | 系統初始化時，建立者即為 Admin（超級管理員） |
| R4 | 角色指派 | 管理員可在後台設定某帳號為 User 或 Guest |
| R5 | 投影片建立/編輯 | Admin 與 User 可在網頁上原生編輯投影片；Guest 不可編輯 |
| R6 | 簡報密碼保護 | 每份簡報可設定獨立的瀏覽密碼 |
| R7 | 瀏覽追蹤 | 記錄「誰、在何時、瀏覽了哪份簡報、第幾頁」 |
| R8 | IP 記錄 | 記錄所有瀏覽行為的來源 IP 位址 |
| R9 | 安全性 | 認證、授權、輸入驗證、XSS/CSRF/注入防護等 |
| R10 | 美觀 | 現代化、專業、響應式的 UI |
| R11 | 公開簡報藝廊 | 提供一個可瀏覽**全部公開簡報**的頁面（跨所有作者），任何使用者皆可探索與觀看 |

### 1.3 設計決策（已確認）
- **編輯模式：Markdown**。投影片以 **Markdown 原始碼**撰寫，使用 `---`（獨立一行）分頁。編輯器為「左側原始碼 + 右側即時預覽」的簡潔介面（取代原本的畫布元素編輯器）。Admin/User 直接寫 Markdown，瀏覽時即時渲染為 HTML。
- **簡報來源（兩種）**：
  - **Markdown 原生**（`sourceType = MARKDOWN`）：在網頁上以 Markdown 建立／編輯，**可編輯**。
  - **PPTX 匯入**（`sourceType = PPTX`）：上傳 `.pptx`，後端以 LibreOffice 轉成每頁圖片，**唯讀**瀏覽（無法在網頁編輯，需重新上傳替換）。
- **上傳功能（右側面板）**：Admin/User 可上傳 **Markdown 檔**（`.md`／`.markdown`，直接匯入成可編輯簡報）或 **PowerPoint 檔**（`.pptx`，轉圖唯讀）。Guest 不可上傳。
- **匿名瀏覽公開內容**：**未登入的訪客即可瀏覽公開簡報**（`PUBLIC`）與首頁公開藝廊（`/`），**不需登入**。其餘（私人、密碼、不公開、建立/編輯/上傳、後台）仍須登入與授權。此處「匿名訪客」指**未登入者**，與「Guest 角色」（已登入但未升級）不同。
- **技術棧**：Next.js 全端（理由見 §3）。
- **身分驗證**：同時支援 **Email + 密碼** 與 **Google OAuth** 兩種登入方式。

### 1.4 名詞定義
- **Deck（簡報集）**：一份完整的簡報，由多張 Slide 組成。對應「一份 PPT」。每份 Deck 有 `sourceType`（`MARKDOWN` 或 `PPTX`）。
- **Slide（投影片）**：簡報集中的單一頁面。內容為 **Markdown 片段**（`MARKDOWN` 來源）或 **一張圖片**（`PPTX` 來源，由 `.pptx` 轉換而來）。
- **Slide 分隔符**：Markdown 中以獨立一行的 `---` 分頁（Marp／reveal.js 慣例）。
- **View（瀏覽記錄）**：一次瀏覽行為的稽核記錄。

## 2. 角色與權限模型

### 2.1 角色定義

| 角色 | 說明 | 取得方式 |
|------|------|----------|
| **Admin（管理員）** | 系統最高權限。可管理所有使用者、所有簡報、查看所有稽核記錄。 | 系統首位註冊者自動成為 Admin；之後可由現有 Admin 指派。 |
| **User（使用者）** | 可建立、編輯、刪除「自己的」簡報；可瀏覽有權限的簡報。 | 由 Admin 指派。 |
| **Guest（訪客）** | 唯讀。僅能瀏覽被授權或公開的簡報，不可建立/編輯/刪除。 | 預設新註冊者為 Guest（待 Admin 指派升級）。 |

> **預設角色策略**：新註冊帳號預設為 `GUEST`，需由 Admin 在後台手動升級為 `USER` 或 `ADMIN`。此為「最小權限原則」(Principle of Least Privilege)，避免任何人註冊即可上傳內容。

### 2.2 權限矩陣（RBAC）

| 操作 | Admin | User | Guest | 匿名（未登入） |
|------|:-----:|:----:|:-----:|:-----:|
| 註冊 / 登入 | ✅ | ✅ | ✅ | ✅ |
| 瀏覽首頁公開藝廊 `/` | ✅ | ✅ | ✅ | ✅ |
| 瀏覽公開簡報（`PUBLIC`） | ✅ | ✅ | ✅ | ✅ |
| 瀏覽受保護簡報（需密碼/授權） | ✅ | ✅（需符合授權） | ✅（需符合授權） | ❌ |
| 建立簡報 | ✅ | ✅ | ❌ | ❌ |
| 編輯**自己的**簡報 | ✅ | ✅ | ❌ | ❌ |
| 編輯**他人的**簡報 | ✅ | ❌ | ❌ | ❌ |
| 刪除**自己的**簡報 | ✅ | ✅ | ❌ | ❌ |
| 刪除**他人的**簡報 | ✅ | ❌ | ❌ | ❌ |
| 設定簡報密碼 | ✅ | ✅（自己的） | ❌ | ❌ |
| 管理使用者角色 | ✅ | ❌ | ❌ | ❌ |
| 查看全站瀏覽稽核記錄 | ✅ | ✅（僅自己簡報的記錄） | ❌ | ❌ |
| 查看 IP 記錄 | ✅ | ✅（僅自己簡報的記錄） | ❌ | ❌ |

> **權限檢查雙層防護**：前端依角色隱藏 UI（體驗），**但所有權限必須在後端 API 層再次強制驗證**（安全）。前端隱藏不等於安全。

### 2.3 簡報層級的存取控制
每份 Deck 有獨立的可見性設定：
- `PRIVATE`：僅擁有者與 Admin 可見。
- `PASSWORD`：任何擁有連結 + 正確密碼者可瀏覽。
- `PUBLIC`：**任何人皆可瀏覽，含未登入的匿名訪客**；同時出現在首頁公開藝廊（`/`）。
- `UNLISTED`（選配）：知道連結即可瀏覽（含匿名），不在列表/藝廊中顯示。

## 3. 技術棧選型與理由

### 3.1 推薦技術棧

| 層級 | 技術 | 理由 |
|------|------|------|
| 框架 | **Next.js 15（App Router）** | 前後端一體，React Server Components + API Routes，減少維護兩個專案的成本；SSR 利於 SEO 與首屏速度。 |
| 語言 | **TypeScript** | 型別安全，降低執行期錯誤，編輯器與 API 的資料結構共用型別。 |
| UI | **原生 CSS 設計 token（CSS custom properties）+ `next/font`** | 以 `:root` 語意化 token 驅動色彩/字級/間距/陰影（設計系統見 §10）；字體自架載入。無框架黑箱、體積小、完全可控。 |
| 資料庫 | **PostgreSQL** | 關聯式資料庫，適合使用者/簡報/稽核這類強關聯結構；支援 JSONB 儲存投影片內容。 |
| ORM | **Prisma** | 型別安全的資料存取、自動 migration、開發體驗佳。 |
| 認證 | **Auth.js (NextAuth v5)** | 原生支援 Credentials（Email/密碼）+ Google OAuth，Session 管理成熟。 |
| 投影片編輯器 | **Markdown 編輯器**（CodeMirror 6 原始碼 + 即時預覽） | 撰寫單純、體驗清爽；以 `---` 分頁，所見即所得的預覽。 |
| Markdown 渲染 | **markdown-it**（或 marked）+ **DOMPurify** | 將 Markdown 轉 HTML 並**白名單消毒**，防 XSS（見 §9.3）。 |
| PPTX 轉檔 | **LibreOffice（headless）/ `soffice`** | 將上傳的 `.pptx` 轉為 PDF→每頁 PNG（或直接轉圖），作為唯讀投影片。於獨立容器或同 image 內執行。 |
| 檔案儲存（圖片/簡報資源） | **S3 相容物件儲存**（Zeabur MinIO / Cloudflare R2） | 投影片內圖片、PPTX 轉出的頁面圖不放資料庫，改放物件儲存並存 key。 |
| 部署 | **Docker / docker-compose** | 容器化打包 Next.js + PostgreSQL + 物件儲存，環境一致、可移植至任何支援 Docker 的平台（自架/雲端 VM/Zeabur）。詳見 §13。 |

### 3.2 為什麼是 Next.js 全端而非前後端分離？
- 本專案規模屬中小型，前後端分離會帶來雙重部署、CORS、型別不同步等額外成本。
- Next.js 的 Server Actions / API Routes 足以承載本專案的 CRUD 與認證需求。
- 單一 repo、單一部署、型別共用，開發與維護成本最低，最符合「快速做出安全又美觀的成品」的目標。

### 3.3 為什麼編輯用 Markdown，而 `.pptx` 走轉圖唯讀？
- **編輯用 Markdown**：在瀏覽器內重現 PowerPoint 的所見即所得編輯極為複雜且笨重。Markdown 撰寫快、純文字易於版本控制與權限控管，渲染與消毒都單純可控，使用者體驗清爽。每張投影片 = 一段 Markdown，以 `---` 分頁。
- **`.pptx` 走轉圖唯讀**：使用者既有的 PowerPoint 檔保留了精細排版，硬轉成 Markdown 會嚴重失真。因此 `.pptx` 上傳後以 LibreOffice 轉成**每頁圖片**，純粹作為**唯讀**瀏覽；要修改請於 PowerPoint 改後重新上傳，或改用 Markdown 原生簡報。
- **兩者並存**：`Deck.sourceType` 區分 `MARKDOWN`（可編輯）與 `PPTX`（唯讀）。瀏覽器播放器對兩者一致地逐頁切換，瀏覽稽核（`slideOrder`）也一致。
- **Markdown 中的圖片**：可貼 HTTPS 圖片網址，或透過 §14 的物件儲存上傳後以 key 引用。

### 4.2 請求流程範例：瀏覽一張投影片
1. 訪客（可能未登入）開啟 `/d/{deckId}`。
2. Middleware 依 Deck 可見性判斷：
   - `PUBLIC`／`UNLISTED`：**允許匿名**，無需登入。
   - `PRIVATE`：需登入且為擁有者/Admin，否則導向登入。
   - `PASSWORD`：未持有效憑證時導向密碼頁。
3. 若 Deck 為 `PASSWORD`，先驗證密碼（成功後發短期 token cookie）。
4. Server 載入 Deck 與第 N 張 Slide，渲染為唯讀模式。
5. 同時非同步寫入一筆 `ViewLog`（user 可為 null＝匿名、deck、slide、IP、UA、時間）。
6. 回傳頁面；前端的「下一頁」按鈕僅切換 slide index，每次切換再寫一筆瀏覽記錄（或彙整後批次寫入，見 §11）。

### 4.3 PPTX 轉檔流程（上傳 .pptx）
```
Admin/User 上傳 .pptx
   │ 1. 經後端驗證（類型/大小/權限）後存入物件儲存 tmp/
   ▼
後端建立 Deck(sourceType=PPTX, status=PROCESSING)
   │ 2. 觸發轉檔工作（同步小檔 or 背景佇列大檔）
   ▼
LibreOffice headless：soffice --headless --convert-to pdf  → pdftoppm/pdftocairo 轉每頁 PNG
   │ 3. 逐頁圖片上傳物件儲存 decks/{deckId}/pages/{n}.webp
   ▼
為每頁建立 Slide(content={kind:"image", src:key})，Deck.status=READY
```
- **同步 vs 背景**：小檔可在請求內同步轉換；較大檔或高併發建議改為背景佇列（如以 BullMQ/Redis 或簡單的 DB 任務表）並回報進度，避免阻塞請求與逾時。
- **安全隔離**：LibreOffice 轉檔在受限環境執行（資源/時間上限、無網路、非 root），詳見 §9.6。

## 5. 資料庫設計

### 5.1 ER 概觀
```
User 1───* Deck 1───* Slide
User 1───* ViewLog *───1 Deck
Deck 1───* DeckAccess（密碼通過或授權記錄，選配）
User 1───* Account（OAuth 綁定，Auth.js）
User 1───* Session（Auth.js）
```

### 5.2 Prisma Schema（草案）

```prisma
// 角色列舉
enum Role {
  ADMIN
  USER
  GUEST
}

// 簡報可見性
enum Visibility {
  PRIVATE    // 僅擁有者與 Admin
  PASSWORD   // 需密碼
  PUBLIC     // 所有登入者
  UNLISTED   // 知道連結即可
}

// 簡報來源
enum SourceType {
  MARKDOWN   // Markdown 原生，可在網頁編輯
  PPTX       // 由 .pptx 轉圖匯入，唯讀
}

// 簡報狀態（主要供 PPTX 轉檔使用）
enum DeckStatus {
  READY        // 可瀏覽
  PROCESSING   // 轉檔中
  FAILED       // 轉檔失敗
}

model User {
  id            String     @id @default(cuid())
  email         String     @unique
  name          String?
  image         String?
  passwordHash  String?    // Email 登入用；OAuth-only 使用者為 null
  role          Role       @default(GUEST)
  isActive      Boolean    @default(true)
  createdAt     DateTime   @default(now())
  updatedAt     DateTime   @updatedAt

  decks         Deck[]
  viewLogs      ViewLog[]
  accounts      Account[]  // Auth.js OAuth
  sessions      Session[]  // Auth.js
}

model Deck {
  id            String     @id @default(cuid())
  title         String
  description   String?
  ownerId       String
  owner         User       @relation(fields: [ownerId], references: [id], onDelete: Cascade)
  visibility    Visibility @default(PRIVATE)
  sourceType    SourceType @default(MARKDOWN)  // MARKDOWN（可編輯）/ PPTX（唯讀）
  status        DeckStatus @default(READY)     // PPTX 轉檔中為 PROCESSING
  sourceFile    String?    // PPTX 來源檔的物件儲存 key（保留以利重轉/下載，選配）
  passwordHash  String?    // 設定密碼時填入（bcrypt/argon2 雜湊）
  coverImage    String?
  createdAt     DateTime   @default(now())
  updatedAt     DateTime   @updatedAt

  slides        Slide[]
  viewLogs      ViewLog[]

  @@index([ownerId])
  @@index([visibility])
}

model Slide {
  id        String   @id @default(cuid())
  deckId    String
  deck      Deck     @relation(fields: [deckId], references: [id], onDelete: Cascade)
  order     Int      // 投影片順序
  content   Json     // { kind:"markdown", markdown } 或 { kind:"image", src, alt }（見 §6.4 / 附錄 B）
  notes     String?  // 講者備註（選配）
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@unique([deckId, order])
  @@index([deckId])
}

model ViewLog {
  id         String   @id @default(cuid())
  deckId     String
  deck       Deck     @relation(fields: [deckId], references: [id], onDelete: Cascade)
  userId     String?  // 未登入瀏覽者為 null（若允許匿名）
  user       User?    @relation(fields: [userId], references: [id], onDelete: SetNull)
  slideOrder Int?     // 瀏覽到第幾頁
  ipAddress  String   // 來源 IP（IPv4/IPv6）
  userAgent  String?
  referer    String?
  viewedAt   DateTime @default(now())

  @@index([deckId])
  @@index([userId])
  @@index([viewedAt])
  @@index([ipAddress])
}

// Auth.js 標準表（Account / Session / VerificationToken）依官方 schema 建立
```

> **重要**：`passwordHash`（無論使用者密碼或簡報密碼）一律以 **argon2id** 或 **bcrypt(cost≥12)** 雜湊儲存，**絕不**明文。

### 5.3 索引與效能
- `ViewLog` 將是成長最快的表，對 `deckId`、`userId`、`viewedAt`、`ipAddress` 建索引以利稽核查詢。
- 長期可考慮對 `ViewLog` 做分區（依月份）或定期歸檔。

## 6. 功能規格

### 6.1 認證（Authentication）
- **Email + 密碼**
  - 註冊：Email 格式驗證、密碼強度規則（至少 10 碼，含大小寫與數字）、密碼以 argon2id 雜湊。
  - 登入：失敗次數限制 + 速率限制（防暴力破解）。
  - 選配：Email 驗證信、忘記密碼流程。
- **Google OAuth**
  - 透過 Auth.js Google Provider。
  - 首次以 Google 登入者自動建立 User（角色 `GUEST`）。
  - 同 Email 帳號的綁定策略需定義（建議：相同 email 視為同一人，需驗證）。
- **Session**：HttpOnly + Secure + SameSite=Lax Cookie；JWT 或 DB Session 皆可（建議 DB Session 便於強制登出）。

### 6.2 角色管理（Admin 後台）
- 使用者列表：搜尋、分頁、顯示角色與狀態。
- 變更角色：`GUEST ⇄ USER ⇄ ADMIN`。
- 停用/啟用帳號（`isActive`）。
- **保護機制**：系統至少保留一名 Admin（不可把最後一位 Admin 降級）；Admin 不可停用自己。

### 6.3 簡報管理（Deck CRUD）
- 建立簡報（標題、描述、可見性）。
- 列表：Admin 看全部；User 看自己的 + 有權限的；Guest 看公開/授權的。
- 編輯簡報設定（標題、可見性、密碼）。
- 刪除簡報（軟刪除選配；硬刪除需二次確認）。

### 6.4 投影片編輯器（Markdown，核心）
> 僅適用 `sourceType = MARKDOWN` 的簡報；`PPTX` 簡報為唯讀，不顯示編輯器。

- **介面**：左側 Markdown 原始碼（CodeMirror 6），右側即時預覽。頂部顯示頁碼／儲存狀態。**整份 Deck 為單一 Markdown 文件**，以獨立一行的 `---` 分頁。
- **資料表示**：每張 Slide 的 `content` 為 `{ "kind": "markdown", "markdown": "# 標題\n\n- 重點一\n- 重點二" }`。儲存時後端把整份文件依 `---` 切段，**同步**到各 Slide 列（新增/更新/刪除/重排），維持 `slideOrder` 與瀏覽稽核一致。
- **支援語法**：標題、清單、粗體/斜體、連結、程式碼區塊、引言、表格、圖片（`![](https URL 或上傳的 key)`）。可選擇支援有限的前置設定（如 `# ` 視為標題頁置中）。
- **Markdown 範例**：
```markdown
# 第一頁標題
歡迎使用 SlideForge

---

## 第二頁
- 重點一
- 重點二

![示意圖](https://.../img.webp)
```
- **權限**：僅 Admin 與簡報擁有者（User）可進入編輯模式；Guest 無編輯入口且後端拒絕其寫入。
- **儲存**：自動存檔（debounce）或手動存檔；每次寫入後端皆以 Zod 驗證內容與長度上限（投影片數、單頁字數）。
- **重要安全點**：Markdown 轉 HTML 後**必須**經 DOMPurify 白名單消毒再渲染；嚴禁未消毒的 `dangerouslySetInnerHTML`（見 §9.3）。預覽（client）與瀏覽（server/client）皆須消毒。

### 6.5 瀏覽播放器（Viewer）
- 全螢幕/置中舞台，固定長寬比（如 16:9）。
- 控制項：上一頁、下一頁、頁碼指示（如 `3 / 12`）、鍵盤左右鍵、全螢幕。
- **渲染**：依 Slide `content.kind`——
  - `markdown`：以 markdown-it 轉 HTML → DOMPurify 消毒 → 渲染（套用投影片排版樣式）。
  - `image`：顯示該頁圖片（PPTX 轉出的頁面）。
- 兩種來源共用同一套頁面切換與控制列，體驗一致。
- **進場動畫**：每次換頁時，投影片淡入、內容**逐項浮現**（stagger）；以換頁重新觸發（播放器對 SlideView 用 `key={index}` 重掛載）。編輯器預覽維持靜態以利對齊。全程**尊重 `prefers-reduced-motion`**（見 §6.9 / §10）。
- 每次切換頁面寫入瀏覽記錄。

### 6.6 簡報密碼保護
- 擁有者可為 Deck 設定/移除密碼（雜湊儲存）。
- 瀏覽 `PASSWORD` 類型 Deck 時出現密碼輸入頁；驗證成功後發放短期憑證（HttpOnly Cookie，綁定該 deckId，有效期如 2 小時）。
- 密碼驗證亦受速率限制保護。

### 6.7 上傳簡報（右側面板）
> 入口位於儀表板／編輯區的**右側面板**。權限：Admin 與 User 可上傳；**Guest 不可**（前端隱藏入口 + 後端 403）。

支援兩種上傳，皆先建立一份新的 Deck：

| 類型 | 副檔名 | 結果 | 可編輯 |
|------|--------|------|:------:|
| **Markdown 匯入** | `.md`、`.markdown` | 讀取檔案內容，依 `---` 切成多張 Slide，建立 `sourceType=MARKDOWN` 的簡報 | ✅ 可在 §6.4 編輯器續編 |
| **PowerPoint 匯入** | `.pptx` | 上傳→LibreOffice 轉每頁圖片→建立 `sourceType=PPTX` 的簡報（§4.3） | ❌ 唯讀，需重新上傳替換 |

- **Markdown 匯入**：純文字、輕量；上傳後即可瀏覽與編輯。需限制檔案大小與投影片數上限，並在渲染時消毒（§9.3）。
- **PPTX 匯入**：上傳後 Deck 進入 `PROCESSING`，轉檔完成轉 `READY`；失敗為 `FAILED` 並提示重試。轉檔細節與安全見 §4.3 / §9.6。
- **限制**：副檔名 + MIME + magic bytes 三重檢查；大小上限（如 Markdown 1MB、PPTX 50MB，可調）；單份簡報頁數上限。

#### 更新既有簡報（上傳覆蓋）
除了「新建」，亦可**上傳檔案覆蓋既有簡報的內容**（入口：工作區卡片的「更新」，限擁有者/Admin）：
- **Markdown 簡報** → 上傳 `.md`／`.markdown`，整份覆蓋所有投影片（重用 `PUT /api/decks/:id/markdown`）。
- **PPTX 簡報** → 上傳 `.pptx`，重新轉檔並取代所有頁面（`POST /api/decks/:id/import/pptx`）。
- 後端以擁有權/角色把關（防 IDOR），並檢查簡報類型需與上傳檔相符；PPTX 更新期間同樣 `PROCESSING → READY/FAILED`。

### 6.8 公開簡報藝廊（首頁）
> 路徑 `/`（**首頁即藝廊**）。提供一個可瀏覽**全部公開簡報**的頁面（跨所有作者），與「我的簡報」（`/dashboard`，僅自己＋有權限者）區隔開來。舊路徑 `/explore` 保留並自動轉址至首頁（含 `q`／`sort` 參數），維持既有連結相容。

- **顯示範圍**：僅列出 `visibility = PUBLIC` 且 `status = READY` 的簡報；**不含** `PRIVATE`／`PASSWORD`／`UNLISTED`（即使是 Admin，此頁也只呈現公開內容，避免私密簡報外洩）。
- **誰可看**：**任何人皆可進入此頁，含未登入的匿名訪客**（不需登入）。已登入者（Admin/User/Guest）亦同。點入播放的也只會是公開簡報。
- **介面**：卡片網格（封面、標題、作者、頁數、瀏覽數、來源徽章 Markdown/PPTX）；支援**關鍵字搜尋**與**排序**（最新／最多瀏覽）；分頁或無限捲動。
- **互動**：點卡片 → `/d/:id` 播放（唯讀）。Guest 同樣只能瀏覽、不能編輯。
- **瀏覽追蹤**：經此頁進入播放一樣寫入 `ViewLog`（含 IP），與 §11 一致。
- **導覽列**：在主導覽加入「探索 / Explore」入口，所有登入者可見。

### 6.9 簡報主題與動畫（`s-*` 安全版面詞彙）
> 目標：在**不放寬安全消毒**（不允許 `<style>`／inline CSS／JS）的前提下，讓 Markdown 簡報能做出旗艦級精美版面與動畫。

#### 核心機制（為何安全）
- DOMPurify 消毒**會移除** `<style>`、inline `style=`、事件屬性與 JS；但**保留** `<div class="…">`、`<span class="…">` 與安全標籤。
- 因此「精美版面」透過一組**預先在 `app/globals.css` 定義好的白名單 CSS 類別（`s-*`）**達成，而非作者自由寫 CSS。作者只能「引用」既有類別，無法注入樣式或腳本 → **零 XSS 風險**（呼應 §9.3）。

#### `s-*` 版面詞彙（節錄）
| 類別 | 用途 |
|------|------|
| `s-slide` | 全幅投影片容器（垂直置中）；每張投影片包一層 |
| `s-dark` / `s-light` | 深 / 淺主題（疊在 `s-slide`） |
| `s-center` | 內容置中 |
| `s-kicker` / `s-chapter` | 小標／章節標籤 |
| `s-lead` | 大字副標 |
| `s-gradient` | 漸層文字（`<span>` 包關鍵字） |
| `s-pill` | 圓角標籤 |
| `s-cols` / `s-cols-3` | 兩欄 / 三欄網格 |
| `s-card` / `s-metric` / `s-quote` | 卡片 / 數據卡 / 引言框 |
| `s-orb` `s-orb-a/-b` | 裝飾光球（緩慢浮動，放在投影片內容最後） |

#### 撰寫規則（重要）
- 投影片之間用單獨一行的 `---` 分頁；投影片內**不可**出現獨立的 `---`。
- **包裹用的 `<div>` 前後必須留空行**，Markdown 才會被解析（CommonMark：HTML 區塊遇空行結束，中間內容才當 Markdown 處理）。範例：
```markdown
<div class="s-slide s-dark s-center">

<div class="s-kicker">INTRODUCING</div>

# 標題可用 <span class="s-gradient">漸層</span>

<p class="s-lead">副標文字</p>

</div>
```
- **純文字 Markdown（不包 `s-slide`）也支援**，套用預設主題並同樣有進場動畫。
- 範例檔見 `docs/examples/`（純文字 01–04）與 `docs/examples/animated/`（旗艦 `s-*` 範例 + 撰寫說明）。

#### 動畫
- 由播放器套用（`SlideView` 的 `animate` 旗標）：投影片淡入 + 內容逐項浮現（CSS keyframe + `nth-child` stagger）；光球持續浮動。編輯器預覽不套動畫。
- **一律受 `prefers-reduced-motion` 規範**（系統開啟減少動態時自動降為靜態）。

## 9. 安全性設計

> 安全性是本專案明確要求，以下為**必做**清單。

### 9.1 認證與 Session
- 密碼以 **argon2id**（或 bcrypt cost≥12）雜湊，加鹽，永不明文。
- Session Cookie：`HttpOnly`、`Secure`、`SameSite=Lax`。
- 登入/註冊/密碼驗證端點皆套用**速率限制**（如每 IP 每分鐘 N 次）與帳號鎖定，防暴力破解。
- 強制 HTTPS（HSTS）。

### 9.2 授權（Authorization）
- **後端為唯一信任邊界**：每個 API 都重新驗證「此 Session 的角色/擁有權是否允許此操作」。前端隱藏僅為體驗。
- 物件層級授權：操作 Deck/Slide 前確認 `ownerId === session.userId || role === ADMIN`。
- 防 IDOR：所有以 `:id` 取資源的端點都檢查存取權，不可只靠「猜不到 ID」。
- **匿名（未登入）存取採白名單**：middleware 只對**明確的公開路由**放行匿名——`/`、`/login`、`/register`、`/explore`、`/d/:id`（及其密碼頁）、`GET /api/decks/public`、`GET /api/decks/:id`、`POST /api/decks/:id/view`、`POST /api/decks/:id/verify-password`。其餘一律要求登入。
- **公開讀取端點仍須在後端按 Deck 可見性把關**：匿名只能取 `PUBLIC`/`UNLISTED`（或通過密碼）的 Deck；`PRIVATE`、未通過密碼的 `PASSWORD`、以及任何 slides 內容，匿名一律 403。`/api/decks/public` 只回傳 `PUBLIC + READY`，並**剝除** `passwordHash` 等敏感欄位。
- 匿名可達的端點同樣套**速率限制**（防爬取/濫用）。

### 9.3 輸入驗證與輸出編碼
- 所有輸入以 **Zod** 驗證（型別、長度、範圍、白名單）。
- 投影片 `content` JSON 在寫入前以 schema 嚴格驗證，拒絕未知欄位。
- **Markdown XSS（重點）**：Markdown 會被使用者撰寫/匯入，渲染成 HTML 後**必須**經 **DOMPurify** 白名單消毒才可輸出。
  - markdown-it 設定 `html: false`（停用內嵌原始 HTML）或在轉出後嚴格 sanitize。
  - 連結 `href`、圖片 `src` 僅允許 `https:`（與物件儲存 key），封鎖 `javascript:`、`data:` 等危險協定。
  - **預覽（編輯器 client）與正式瀏覽（Viewer）皆須消毒**——不可只信任其中一端。
  - 嚴禁把未消毒結果丟入 `dangerouslySetInnerHTML`。
  - **簡報主題（§6.9）不放寬消毒**：作者只能引用預定義的 `s-*` 白名單 CSS 類別（保留 `class`），仍**移除** `<style>`／inline `style`／JS／事件屬性 → 精美版面與 XSS 防護並存。
- 防 **SQL 注入**：一律透過 Prisma 參數化查詢，不拼接 SQL。

### 9.4 CSRF
- Auth.js 內建 CSRF 保護；自訂的狀態變更端點使用 SameSite Cookie + CSRF token 或僅接受同源請求。

### 9.5 內容安全政策（CSP）與標頭
- 設定 CSP（限制 script 來源）、`X-Content-Type-Options: nosniff`、`X-Frame-Options: DENY`（防點擊劫持，瀏覽器頁可視需求調整）、`Referrer-Policy`。

### 9.6 檔案上傳安全
- 類型白名單 + 副檔名/MIME/**magic bytes** 三重檢查 + 大小上限。
- 圖片重新編碼（去除惡意 payload / EXIF）。
- 上傳檔不可執行；以隨機檔名儲存於物件儲存，不放於可執行路徑。
- **Markdown 檔匯入**：以純文字讀取，**不執行**其中任何內容；渲染時走 §9.3 消毒；限制檔案大小與投影片數。
- **PPTX 轉檔隔離**：LibreOffice/`soffice` 轉檔是高風險的解析動作，須在受限環境執行——
  - 獨立 worker/容器，**非 root**、唯讀檔案系統 + 專用暫存目錄、**無對外網路**。
  - 設 CPU/記憶體/逾時上限，避免惡意檔造成 DoS（zip bomb、無限轉檔）。
  - 限制單檔大小與頁數上限；轉檔產物（圖片）一律經 §14.6 重新編碼。
  - 轉檔失敗或逾時 → Deck 標記 `FAILED`，清理暫存，不留半成品。

### 9.7 密碼保護的簡報
- 簡報密碼雜湊儲存；驗證採固定時間比較。
- 通過後發短期 scoped cookie，不在 URL 帶密碼。

### 9.8 隱私與 IP 記錄合規
- 記錄 IP/UA 屬個資範疇：應於隱私政策揭露用途、保存期限（如 180 天）、與存取權限（僅 Admin）。
- 取得真實 IP 時，於反向代理後需正確解析 `X-Forwarded-For`（且僅信任自家代理鏈），避免 IP 偽造。
- 提供稽核記錄的保存期限與清理排程。

### 9.9 相依套件與機密
- 機密（DB 連線、OAuth secret、Session secret）放環境變數，不入版控。
- 啟用 Dependabot / `npm audit`；CI 加入 secret 掃描（gitleaks）與 SAST。

## 10. UI/UX 與美觀設計

> **設計系統名稱：Editorial SaaS**。token 定義於 `app/globals.css` 的 `:root`；字體以 `next/font` 自架載入。以下為**美感邏輯**與**配色邏輯**的設計準則，供任何擴充頁面時遵循一致性。

### 10.1 設計原則（總綱）
- **內容優先**：介面是配角、投影片是主角。介面用低彩度、留白、輕量結構，把注意力讓給內容。
- **Token 驅動**：所有顏色、字級、間距、圓角、陰影皆為語意化 CSS 變數；元件不寫死數值，改色/做深色模式只動 token。
- **一致性**：同一套 scale 貫穿全站（間距 8pt 節奏、圓角 8/12/16/22、三階陰影）。
- **可及性即設計**：對比達 WCAG AA、鍵盤可操作、可見 focus 環、尊重 `prefers-reduced-motion`、不只用顏色傳達資訊。
- **響應式**：桌面為主，平板/手機可用；mobile-first 斷點 375 / 768 / 1024 / 1440。

### 10.2 美感邏輯（Aesthetic Logic）
專業介面的「好看」不是靠裝飾，而是靠下列可驗證的原則：

1. **視覺層級用「大小／字重／間距／對比」建立，而非顏色**。標題 display 字體 + 大字級 + 收緊字距；正文規則字重；弱化資訊用 `--muted`，不是用更多顏色。
2. **節奏與留白**：以 8pt 為基準的間距系統製造呼吸感；留白用來「分組相關、區隔無關」，而非隨意填空。區塊間距分層（16 / 24 / 32 / 48 / 60）。
3. **克制（Restraint）**：單一品牌 hue + 中性灰階撐起 90% 畫面，強調色稀用。**每個畫面只有一個主要 CTA**，次要動作視覺上退讓（secondary/ghost）。
4. **質感分層（Elevation）**：陰影代表「離使用者多近」——卡片淺、彈窗深；用固定三階陰影 scale，不出現隨機陰影值。圓角同樣成 scale，不混用。
5. **動效有意義**：動畫只用來表達因果與空間連續（hover 浮起、彈窗從中心淡入縮放、頁面切換）；時長 150–220ms、`ease`/spring 曲線、exit 比 enter 快；一律可被 reduced-motion 關閉。
6. **細節即專業度**：數據/數字用 `tabular-nums` 防跳動；圖示用單一線性家族（Lucide）統一筆畫；不用 emoji 當功能圖示；press 狀態不位移版面。

### 10.3 配色邏輯（Color Logic）
配色採「**三層角色 + 60-30-10 + 對比優先 + 語意 token**」：

- **三層角色**
  - **中性基底（Neutral）**：背景、表面、邊框、文字。決定整體「乾淨度」，佔畫面最大面積。
  - **品牌色（Brand）**：單一 hue（indigo）做成 tonal ramp（soft→base→strong→ink），承載品牌識別與主要互動（CTA、連結、focus）。**只用一個 hue 的深淺**，避免多色相打架。
  - **功能色（Functional）**：success / danger / warning，僅在語意正確時出現，且**必搭 icon/文字**（不可只靠顏色，a11y）。
- **60-30-10 比例**：約 60% 中性背景/表面、30% 文字與結構、10% 品牌強調。強調色越省，CTA 越突出。
- **對比優先（WCAG AA）**：每組前景/背景都驗證——正文 ≥ 4.5:1、大字與 UI glyph ≥ 3:1。`--muted (#6A6E80)` on 白 ≈ 5:1。
- **語意化 token，不寫死 hex**：元件只引用 `--brand / --text / --muted / --danger …`；換色、出深色模式時集中一處改。
- **品牌色梯度的用途分工**：`soft` 作淡底（badge、focus 光環、hover 底）、`base` 作主色（CTA、連結）、`strong` 作 hover 加深、`ink` 作淡底上的文字。
- **點綴（Accent）**：violet 僅用於 logo/hero 漸層；amber 僅用於少量高亮，不參與主要互動。

#### 色彩 Token（淺色，已實作）

### 10.4 字體與字級系統（Typography）
- **字體配對**（邏輯：display 有個性、body 高易讀、CJK 完整覆蓋）
  - 標題 `--font-display`：**Plus Jakarta Sans**（幾何人文無襯線，較 Inter 有個性，撐起品牌調性）。
  - 內文 `--font-sans`：**Inter**（拉丁）+ **Noto Sans TC / PingFang TC / Microsoft JhengHei**（繁中 fallback，免額外下載 CJK 大字檔）。
  - 程式碼 `--font-mono`：系統等寬（`ui-monospace…`）。
  - 以 `next/font` 自架載入（`display:swap`，避免 FOIT 與版面跳動）。
- **模組化字級**（比例 ≈1.22，base 16px）：`--fs-xs 13 / sm 14 / base 16 / md 18 / lg 21 / xl 26 / 2xl 33 / 3xl 44`。
- **字重層級**：標題 700–800、卡片標題/按鈕/標籤 600、正文 400；行高正文 1.6、標題 1.1–1.15。
- **可讀性**：正文行長 65–75 字元（手機 35–60）；標題收緊字距（letter-spacing 約 -.02 ~ -.04em）。

### 10.5 間距、圓角、陰影與動效（Tokens）
- **間距**：8pt 節奏（4 / 8 / 12 / 16 / 24 / 32 / 48 / 60）。
- **圓角**：`--r-sm 8 / md 12 / lg 16 / xl 22`（小元件→卡片→面板→大容器）。
- **陰影（三階 elevation）**：`--sh-sm`（卡片靜置）/ `--sh-md`（浮起）/ `--sh-lg`（彈窗、hover 卡片）。
- **動效**：micro-interaction 150–220ms、曲線 `--ease cubic-bezier(.2,.7,.2,1)`；卡片 hover translateY(-3px)、彈窗 `modal-in` 淡入縮放、按鈕按下 translateY(1px)。全站受 §10.1 reduced-motion 規範。

### 10.6 元件與互動狀態
- **按鈕**：primary（品牌實底 + 品牌色陰影）/ secondary（白底細框）/ danger（紅）/ small；具 hover、active(下沉)、disabled(降透明、無陰影)、focus-visible(焦點環) 四態。
- **卡片**：靜置 `--sh-sm`，hover 浮起換 `--sh-lg`；deck 封面用品牌淡色漸層。
- **徽章**：角色色票（ADMIN 紅 / USER 綠 / GUEST 灰）+ 來源（Markdown/PPTX）。
- **輸入框**：focus 用 4px 柔光環（`--brand-soft`）+ 品牌邊框；placeholder 低對比但可見。
- **彈窗**：scrim 50% 黑 + 3px 模糊，足以隔離前景；淡入縮放動效。

### 10.7 關鍵畫面
- **簡報牆／公開藝廊（首頁 `/`）**：縮圖卡片網格，顯示標題、作者、來源徽章（Markdown/PPTX）、瀏覽數；頂部有搜尋框與排序（最新／最熱門）。藝廊只呈現公開簡報。
- **播放器**：極簡深色舞台，底部浮動控制列，左右大範圍點擊區切換頁。
- **編輯器**：左側 Markdown 原始碼、右側即時預覽（投影片樣式）；頂部頁碼/儲存狀態。簡潔、以文字為中心。
- **上傳面板**：儀表板右側面板，拖放或選檔上傳 `.md` / `.pptx`，顯示轉檔進度與結果。
- **Admin 後台**：資料表格 + 篩選 + 角色下拉。

## 11. 瀏覽追蹤與稽核

### 11.1 記錄內容（每筆 ViewLog）

- 篩選維度：使用者、簡報、IP、時間範圍。
- 視圖：時間線、依簡報彙總、依使用者彙總、熱門頁分析。
- 匯出 CSV（選配）。

### 11.5 User 自助分析
- User 可查看「自己簡報」的瀏覽記錄與 IP，但看不到其他人的。

## 13. 部署與維運

> **部署方式：Docker 容器化**。整個應用（Next.js + PostgreSQL + 物件儲存）以 Docker / docker-compose 打包，確保開發、測試、正式環境一致，可部署到任何支援 Docker 的平台（自架主機、Zeabur、雲端 VM 等）。

### 13.1 環境
- **開發**：`docker compose up`（Next.js dev + PostgreSQL + MinIO 一鍵起）。
- **正式**：Docker image 部署至目標平台；資料庫與物件儲存可用容器或託管服務。
- **PPTX 轉檔相依**：負責轉檔的執行環境需安裝 **LibreOffice（`libreoffice`/`soffice`）** 與 **poppler-utils（`pdftoppm`）**。可（A）裝進 web image，或（B）獨立 `converter` 容器/worker（較易隔離與擴展，呼應 §9.6）。Alpine 對 LibreOffice 支援有限，建議轉檔環境用 Debian 基底 image。

### 13.2 容器化架構

```
└──────────────────────────────────────────────────┘
```

### 13.3 Dockerfile（Next.js，多階段建置）

Next.js 需在 `next.config.js` 設定 `output: 'standalone'` 以產生精簡執行檔。

```dockerfile
# ---- 依賴階段 ----
FROM node:20-alpine AS deps
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci

# ---- 建置階段 ----
FROM node:20-alpine AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
RUN npx prisma generate
RUN npm run build

# ---- 執行階段（精簡）----
FROM node:20-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production
# 以非 root 使用者執行（安全）
RUN addgroup -g 1001 nodejs && adduser -u 1001 -G nodejs -S nextjs
COPY --from=builder /app/public ./public
COPY --from=builder --chown=nextjs:nodejs /app/.next/standalone ./
COPY --from=builder --chown=nextjs:nodejs /app/.next/static ./.next/static
COPY --from=builder /app/prisma ./prisma
USER nextjs
EXPOSE 3000
ENV PORT=3000
CMD ["node", "server.js"]
```

> 容器啟動時需先跑 migration：可用 entrypoint 腳本執行 `npx prisma migrate deploy` 再啟動 `server.js`，或於 compose 用獨立 init 服務跑 migration。

### 13.9 GitHub Actions CI/CD（自動建置並推送 Docker image）

> 目標：push 到 `main`（或打 tag）時，GitHub Actions 自動建置 Docker image 並推送到 **Docker Hub**（或 GHCR）。

#### 流程
```
git push (main / tag v*)
        │
        ▼
┌─────────────────────────────────────────────┐
│ GitHub Actions: docker-publish.yml          │
│  1. checkout                                 │
│  2. 登入 Docker Hub（用 Secrets）            │
│  3. setup buildx（多平台/快取）              │
│  4. docker build                             │
│  5. docker push（tag: latest + commit SHA）  │
└─────────────────────────────────────────────┘
        │
        ▼
   Docker Hub: <user>/ppt-web:latest, :<sha>
```

#### 需設定的 GitHub Secrets
| Secret | 說明 |
|--------|------|
| `DOCKERHUB_USERNAME` | Docker Hub 帳號 |
| `DOCKERHUB_TOKEN` | Docker Hub Access Token（**不要用密碼**，於 Docker Hub → Account Settings → Security 產生） |

> 推 GHCR 則改用內建 `GITHUB_TOKEN`，無需額外 secret。

#### Workflow：`.github/workflows/docker-publish.yml`
```yaml
name: Build and Push Docker Image

on:
  push:
    branches: [main]
    tags: ["v*"]
  workflow_dispatch:   # 允許手動觸發

env:
  IMAGE_NAME: ppt-web

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write   # 推 GHCR 時需要
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Log in to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}

      - name: Extract metadata (tags, labels)
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ secrets.DOCKERHUB_USERNAME }}/${{ env.IMAGE_NAME }}
          tags: |
            type=raw,value=latest,enable={{is_default_branch}}
            type=sha,format=short
            type=semver,pattern={{version}}

      - name: Build and push
        uses: docker/build-push-action@v6
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

#### 推 GHCR（GitHub Container Registry）的替代寫法
若想推到 `ghcr.io`（免額外 secret），把登入與 images 改成：
```yaml
      - name: Log in to GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      # metadata-action 的 images 改為：
      #   images: ghcr.io/${{ github.repository }}
```

#### 建議的完整 CI Pipeline（與安全章節呼應）
建議在推送 image 前先跑品質與安全關卡，任一失敗即不推送：
1. **lint + type check + build**（`npm run lint`、`tsc --noEmit`、`npm run build`）
2. **測試**（Vitest 單元/整合）
3. **gitleaks** secret 掃描（呼應 §9.9）
4. **build Docker image**
5. **Trivy** 掃描 image 漏洞（呼應 §13.7）
6. **push**（僅 main/tag 通過上述步驟後）

> 可拆成多個 job 並用 `needs:` 串接：`test → security-scan → build-and-push`，PR 時只跑前段不推送，合併到 main 才推。

## 14. S3 物件儲存設計

> 投影片內的圖片、簡報封面等**二進位資源不存入 PostgreSQL**，改放 S3 相容物件儲存，DB 僅存物件的 key/URL。本章詳述供應商選型、bucket 結構、上傳流程、安全、CORS/政策、程式碼與生命週期管理。與 §7.5（圖片上傳 API）、§9.6（檔案上傳安全）、§13（部署）相互呼應。

### 14.1 為什麼用物件儲存而非存資料庫？
- **效能**：大型二進位資料放 DB 會撐大資料列、拖慢備份與查詢；物件儲存專為大檔與高併發讀取設計。
- **成本與擴充**：物件儲存單價低、可水平擴充，且能直接掛 CDN 加速。
- **直傳能力**：可發 presigned URL 讓瀏覽器**直接上傳到 S3**，不經過 Next.js 伺服器，省頻寬與記憶體。
- **關注點分離**：應用伺服器無狀態，圖片資源獨立於應用容器，利於水平擴展。

### 14.2 供應商選型（S3 相容）

| 供應商 | 定位 | 優點 | 注意事項 |
|--------|------|------|----------|
| **MinIO（自架）** | 開發 + 自架正式 | 開源、可放進 docker-compose（見 §13.4）、完全自有資料 | 需自行維運、備份、擴容 |
| **Cloudflare R2** | 託管，正式推薦 | **免出口流量費（egress）**、S3 相容 API、可接 CDN | 需 Cloudflare 帳號；部分進階 S3 功能不支援 |
| **AWS S3** | 託管，企業級 | 功能最完整、生態成熟 | egress 計費、設定相對繁瑣 |
| **Zeabur 物件儲存 / 其他** | 託管 | 與部署平台整合 | 視平台支援度 |

> **建議策略**：開發與自架用 **MinIO**；正式環境若要省流量費，優先 **Cloudflare R2**。三者皆相容 S3 API，**程式碼一致，只需切換 endpoint 與金鑰**（見 §14.7），避免供應商鎖定。這也回應 §16.2 待決事項第 5 點。

### 14.3 Bucket 結構與物件 Key 命名

採單一 bucket（如 `ppt-assets`），以路徑前綴分類；key 內嵌 `deckId` 便於授權判斷與生命週期清理：

```
ppt-assets/
├── decks/
│   └── {deckId}/
│       ├── cover/{uuid}.webp          # 簡報封面
│       └── slides/{slideId}/{uuid}.webp   # 投影片內圖片
└── tmp/
    └── {userId}/{uuid}.{ext}          # 上傳暫存（未綁定，定期清理）
```

命名原則：
- **隨機檔名（UUID）**：避免可預測、避免覆蓋、避免路徑穿越（呼應 §9.6）。
- **不信任使用者原始檔名**：原始檔名僅供顯示，不用於 key。
- **副檔名以實際處理後格式為準**（統一轉 `.webp`，見 §14.6）。
- **以 `deckId` 為前綴**：刪除整份 Deck 時可用 prefix 批次刪除（見 §14.9）。

### 14.4 存取模型：私有 bucket + presigned URL（推薦）

預設 bucket **設為私有**（不開放匿名讀取），透過短期 presigned URL 控制上傳與下載：

**上傳流程（presigned PUT，瀏覽器直傳）：**
```
瀏覽器                    Next.js API                  S3 / R2 / MinIO
  │                          │                              │
  │ 1. POST /api/uploads/image (檔名,類型,大小,deckId)        │
  │─────────────────────────>│                              │
  │                          │ 2. 驗證權限 + 類型/大小白名單   │
  │                          │ 3. 產生 key + presigned PUT URL│
  │                          │<─────────────────────────────│
  │ 4. { uploadUrl, key }    │                              │
  │<─────────────────────────│                              │
  │ 5. PUT 檔案 (直傳，帶 Content-Type)                       │
  │──────────────────────────────────────────────────────> │
  │ 6. 200 OK                                               │
  │<────────────────────────────────────────────────────── │
  │ 7. PATCH slide content，寫入 key（不存完整 URL）          │
  │─────────────────────────>│                              │
```

**讀取（顯示圖片）兩種策略：**

| 策略 | 做法 | 適用 |
|------|------|------|
| **A. presigned GET** | 渲染前由後端為每個 key 產生短期 GET URL | 圖片需隨 Deck 權限控管（私有/密碼簡報），安全性高 |
| **B. 公開讀取 + CDN** | 將 `decks/*` 設為公開讀取，前面掛 CDN | 圖片本身不機密、追求最大讀取效能與快取 |

> **建議**：v1 採 **策略 A（presigned GET）**，與 §2.3 簡報可見性、密碼保護一致——私有/密碼簡報的圖片不會因為知道 URL 就外洩。若日後有大量公開簡報再評估策略 B + CDN。presigned GET 的有效期建議短（如 15–60 分鐘），渲染時即時簽發。

### 14.5 上傳安全要點（呼應 §9.6）
- **類型白名單**：僅允許 `image/png`、`image/jpeg`、`image/webp`、`image/gif`；同時檢查副檔名與 MIME。
- **以實際內容判斷類型**：用 magic bytes（檔頭）驗證，不只信前端宣告的 `Content-Type`。
- **大小上限**：在簽發 presigned URL 時用條件限制（如 POST policy 的 `content-length-range`），避免超大檔。
- **重新編碼**：上傳後以 `sharp` 重新編碼為 WebP，去除惡意 payload 與 EXIF（含 GPS 等隱私）。
- **隨機 key + 私有 bucket**：物件不可執行、不可列舉、不放於可執行路徑。
- **最小權限金鑰**：應用使用的 access key 僅授予該 bucket 的必要動作（見 §14.8），不要用 root 金鑰。
- **配額**：可對單一使用者/Deck 設總量上限，避免濫用塞爆儲存。

### 14.6 圖片處理（sharp）

上傳後（或在直傳後由後端非同步處理）統一正規化：

```ts
import sharp from "sharp";

// 將原圖轉為標準 WebP，限制最大尺寸並移除 metadata
export async function normalizeImage(input: Buffer) {
  return sharp(input)
    .rotate()                       // 依 EXIF 自動轉正後即丟棄 EXIF
    .resize({ width: 1920, height: 1920, fit: "inside", withoutEnlargement: true })
    .webp({ quality: 82 })
    .toBuffer();
}
```

> 若採「瀏覽器直傳」流程，重新編碼需改由後端**下載→處理→覆寫**，或改為「上傳到 `tmp/` → 後端處理後寫入正式 key」。簡化版可讓上傳先進 API、由後端處理後再 PUT 到 S3（犧牲直傳的頻寬優勢，換取一定能重新編碼）。**v1 建議走「經後端處理」流程以確保每張圖都被消毒。**

**產生 presigned 上傳 URL（API：`POST /api/uploads/image`）：**

```ts
// app/api/uploads/image/route.ts
import { PutObjectCommand } from "@aws-sdk/client-s3";
import { getSignedUrl } from "@aws-sdk/s3-request-presigner";
import { randomUUID } from "node:crypto";
import { z } from "zod";
import { s3, S3_BUCKET } from "@/lib/s3";
import { auth } from "@/auth";

const ALLOWED = { "image/png": "png", "image/jpeg": "jpg", "image/webp": "webp", "image/gif": "gif" } as const;
const MAX_BYTES = 10 * 1024 * 1024; // 10MB

const Body = z.object({
  deckId: z.string().cuid(),
  contentType: z.enum(Object.keys(ALLOWED) as [string, ...string[]]),
  size: z.number().int().positive().max(MAX_BYTES),
});

export async function POST(req: Request) {
  const session = await auth();
  if (!session || session.user.role === "GUEST") return new Response("Forbidden", { status: 403 });

  const parsed = Body.safeParse(await req.json());
  if (!parsed.success) return new Response("Bad Request", { status: 400 });
  const { deckId, contentType } = parsed.data;

  // 後端授權：確認此使用者可寫入該 deck（owner 或 admin）——呼應 §9.2
  await assertCanEditDeck(session, deckId);

  const ext = ALLOWED[contentType as keyof typeof ALLOWED];
  const key = `decks/${deckId}/slides/${randomUUID()}.${ext}`;

  const uploadUrl = await getSignedUrl(
    s3,
    new PutObjectCommand({ Bucket: S3_BUCKET, Key: key, ContentType: contentType }),
    { expiresIn: 300 }, // 5 分鐘
  );

  return Response.json({ uploadUrl, key });
}
```

**渲染時產生 presigned 讀取 URL（策略 A）：**

```ts
// lib/s3.ts（續）
import { GetObjectCommand } from "@aws-sdk/client-s3";

export function getViewUrl(key: string, expiresIn = 1800) {
  return getSignedUrl(s3, new GetObjectCommand({ Bucket: S3_BUCKET, Key: key }), { expiresIn });
}
```

> Slide `content` JSON 中的 `image.src` 建議**只存 key**（如 `decks/abc/slides/xxx.webp`），渲染前再由後端換成 presigned URL。如此可隨時切換供應商、調整存取策略，URL 過期也不影響資料正確性。對應需放寬 §附錄 B 中 `image.src` 的 `.url()` 驗證為「key 或 URL」。

### 14.8 Bucket 政策與 CORS

**最小權限 IAM 政策（應用金鑰僅能操作該 bucket）：**
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:PutObject", "s3:GetObject", "s3:DeleteObject"],
      "Resource": "arn:aws:s3:::ppt-assets/*"
    },
    { "Effect": "Allow", "Action": ["s3:ListBucket"], "Resource": "arn:aws:s3:::ppt-assets" }
  ]
}
```

**CORS（允許瀏覽器直傳 PUT；正式環境把 `AllowedOrigins` 限定為自家網域）：**
```json
[
  {
    "AllowedOrigins": ["https://你的網域"],
    "AllowedMethods": ["PUT", "GET"],
    "AllowedHeaders": ["Content-Type"],
    "ExposeHeaders": ["ETag"],
    "MaxAgeSeconds": 3000
  }
]
```

> bucket 預設 **Block Public Access / 不開匿名讀取**；只透過 presigned URL 存取（策略 A）。若採策略 B 才在 `decks/*` 開公開讀取。

### 14.9 生命週期與孤兒清理
- **刪除 Deck/Slide 時連帶刪物件**：刪 Slide → 刪該 slide 的圖片 key；刪 Deck → 用 `decks/{deckId}/` 前綴批次刪除（`ListObjectsV2` + `DeleteObjects`）。建議在 Deck/Slide 刪除的 service 層以交易+事後清理處理。
- **暫存清理**：對 `tmp/` 設 lifecycle rule，自動刪除 N 天前未綁定的上傳。
- **孤兒掃描**：排程比對「DB 中引用的 key」與「bucket 實際物件」，刪除無引用者（呼應 §13.8 維運）。
- **編輯時換圖**：Markdown 改用新圖後，舊 key 若不再被任何 slide 引用即標記刪除。

```ts
// 刪除整份 Deck 的所有物件（前綴批次刪除）
import { ListObjectsV2Command, DeleteObjectsCommand } from "@aws-sdk/client-s3";

export async function deleteDeckAssets(deckId: string) {
  const prefix = `decks/${deckId}/`;
  let token: string | undefined;
  do {
    const list = await s3.send(new ListObjectsV2Command({ Bucket: S3_BUCKET, Prefix: prefix, ContinuationToken: token }));
    const objects = (list.Contents ?? []).map((o) => ({ Key: o.Key! }));
    if (objects.length) {
      await s3.send(new DeleteObjectsCommand({ Bucket: S3_BUCKET, Delete: { Objects: objects } }));
    }
    token = list.IsTruncated ? list.NextContinuationToken : undefined;
  } while (token);
}
```

### 14.10 本機/開發：MinIO 設定
- docker-compose 已含 `minio` 服務（見 §13.4）。
- 首次啟動需**建立 bucket** 並設 CORS。可用獨立 init 容器跑 `mc`：

```yaml
      mc mb -p local/$$S3_BUCKET &&

      S3_ACCESS_KEY: ${S3_ACCESS_KEY}

      S3_BUCKET: ${S3_BUCKET}
```

> MinIO 主控台預設在 `:9001`，可用於檢視物件與設定政策。正式環境的 MinIO 連接埠不對外公開（呼應 §13.7）。

### 14.11 相依套件
```bash
npm i @aws-sdk/client-s3 @aws-sdk/s3-request-presigner sharp
```
> `sharp` 含原生二進位，於 Alpine 容器須確認可正常安裝（必要時於 Dockerfile 安裝對應系統函式庫，或改用 Debian 基底 image）。

### 14.12 測試重點（併入 §15 測試策略）
- Guest 呼叫 `/api/uploads/image` → 必須 403。
- User A 對 User B 的 deck 取簽名 URL → 必須 403（IDOR 防護）。
- 上傳非白名單類型或超過大小上限 → 拒絕。
- 偽造副檔名但內容非圖片 → magic bytes 檢查擋下。
- 刪除 Deck → 對應 `decks/{deckId}/` 物件全部清除，無孤兒殘留。
- presigned GET URL 過期後不可再存取。

## 15. 測試策略

| 層級 | 範圍 | 工具 |
|------|------|------|
| 單元測試 | 權限服務、密碼雜湊、Zod schema、IP 解析 | Vitest |
| 整合測試 | API 端點（授權矩陣、密碼驗證、CRUD） | Vitest + 測試 DB |
| E2E | 登入→建立簡報→瀏覽→稽核 主流程 | Playwright |
| 安全測試 | 授權繞過、IDOR、XSS、速率限制 | 手動 + 自動掃描 |

**重點測試案例**：
- Guest 嘗試呼叫編輯 API → 必須 403。
- User A 嘗試編輯 User B 的簡報 → 必須 403。
- 連續錯誤密碼 → 觸發速率限制。
- 注入含 `<script>` 的投影片文字 → 渲染時不執行。
- 偽造 `X-Forwarded-For` → 記錄到的仍是可信 IP。

```ts
async function assertCanEditDeck(session, deckId) {
  const deck = await db.deck.findUnique({ where: { id: deckId } });
  if (!deck) throw new NotFound();
  const isOwner = deck.ownerId === session.userId;
  const isAdmin = session.role === 'ADMIN';
```

## 附錄 B：投影片 content JSON Schema（Zod 草案）

每張 Slide 的 `content` 為兩種之一：`markdown`（可編輯簡報）或 `image`（PPTX 轉出的頁面，唯讀）。

```ts
const SlideContentSchema = z.discriminatedUnion('kind', [
  // Markdown 原生投影片（可編輯）
  z.object({
    kind: z.literal('markdown'),
    markdown: z.string().max(20000),     // 單頁 Markdown 原始碼上限
  }).strict(),
  // PPTX 轉出的頁面圖片（唯讀）。src 存物件儲存 key 或 https URL（見 §14.7）
  z.object({
    kind: z.literal('image'),
    src: z.string().max(2048),
    alt: z.string().max(300).optional(),
  }).strict(),
]);

// 匯入/編輯時的整份 Markdown 文件（以 --- 分頁，切段後逐段套上 SlideContentSchema）
const DeckMarkdownSchema = z.object({
  markdown: z.string().max(500000),      // 整份上限
});
```

> **渲染安全**：`markdown` 內容在輸出前一律經 markdown-it（`html:false`）+ DOMPurify 消毒；`image.src` 為 key 時於渲染前換成 presigned URL（§14.7）。對應 §9.3。
